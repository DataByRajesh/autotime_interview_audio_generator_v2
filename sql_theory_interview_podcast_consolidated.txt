TRAINING LOOP 1. CONCEPT THEORY AND INTERVIEW EXPLANATION.



SECTION: SQL.
SQL stands for Structured Query Language.

In simple terms, SQL is the language used to communicate with databases.

A database stores structured business information.

For example, a fintech system may store customers, accounts, payments, transactions, settlements, audit logs, support tickets, and reports.

SQL helps us ask questions such as:
Which customer is affected?
Which payment failed?
When did the status change?
How many records are impacted?
Is a value missing?
Is a record duplicated?
Did the system update happen correctly?

For your target roles, SQL is not just a developer skill.
It is a validation and investigation skill.

Interview answer:
SQL is used to retrieve, filter, update, and analyse data stored in relational databases. In analyst roles, I mainly use SQL to validate data, investigate issues, support UAT, and understand system behaviour.

Rewind:
SQL helps me understand the truth inside the system.




SECTION: DATABASE.
A database is an organised place where applications store and manage data.

When a customer uses a banking application, they see screens and buttons.
Behind those screens, the system stores data in a database.

The application may read customer details from the database.
It may write payment records to the database.
It may update transaction statuses.
It may store audit logs for compliance.

For a Business Systems Analyst, the important point is not only that a database exists.
The important point is that business workflows are represented as data.

A payment workflow becomes payment records.
A customer onboarding process becomes customer and account records.
A user action becomes an audit log record.

Interview answer:
A database stores structured business data. SQL helps me retrieve and validate that data to understand whether the system is working as expected.

Rewind:
A database stores business activity in structured form.




SECTION: TABLE.
A table is a structured collection of related records.

It is similar to a spreadsheet, but more controlled.

Each table usually represents one business entity or one business concept.

Examples:
customers table,
accounts table,
payments table,
transactions table,
settlements table,
audit logs table.

A customers table stores customer data.
A payments table stores payment data.
An audit logs table stores system action history.

Interview answer:
A table stores data about a specific business entity, such as customers, accounts, payments, or transactions.

Business analyst angle:
Understanding tables helps me understand how a business process is represented inside the system.

Rewind:
Tables organise business entities.




SECTION: ROW.
A row is one individual record in a table.

In a customers table, one row may represent one customer.
In a payments table, one row may represent one payment.
In a transactions table, one row may represent one transaction.

When investigating issues, analysts often start with a row.

For example:
One failed payment.
One customer record.
One transaction.
One support ticket.
One audit log entry.

Interview answer:
A row represents one record in a table.

Analyst answer:
When investigating an issue, I identify the affected row or rows and check important fields such as status, amount, timestamp, and related IDs.

Rewind:
A row is one record.




SECTION: COLUMN.
A column is a field or attribute in a table.

For example, a payments table may have:
payment ID,
customer ID,
amount,
currency,
payment status,
created date,
updated date.

Each column has business meaning.

Payment status tells whether the payment is pending, completed, failed, or cancelled.
Amount tells the payment value.
Created date tells when the record was created.
Updated date tells when it changed.

Interview answer:
A column represents a specific attribute of a record.

Analyst answer:
Columns matter because each field can represent a business rule, system status, value, date, or control point.

Rewind:
Columns are fields with business meaning.




SECTION: DATA TYPE.
A data type defines what kind of value a column can store.

Common data types include text, number, decimal, date, timestamp, and boolean.

Why does this matter?

A payment amount should be numeric.
A date field should store a valid date or timestamp.
A status field should store controlled text values.
A true or false field should store a boolean value.

If data types are wrong, reporting and calculations can fail.

In fintech, this matters because financial reporting, payment reconciliation, and compliance checks depend on clean data.

Interview answer:
A data type controls the kind of value a column can store. Correct data types help maintain data quality and reliable system behaviour.

Rewind:
Data types protect data quality.




SECTION: PRIMARY KEY.
A primary key uniquely identifies each row in a table.

For example:
customer ID identifies one customer.
payment ID identifies one payment.
transaction ID identifies one transaction.

Names can repeat.
Emails can change.
Phone numbers can change.
But a primary key should uniquely identify the record.

Interview answer:
A primary key is a unique identifier for each record in a table.

Fintech example:
A payment ID helps track one specific payment from initiation to completion or failure.

Rewind:
A primary key gives each row a unique identity.




SECTION: FOREIGN KEY.
A foreign key connects one table to another.

For example:
The customers table has customer ID.
The payments table may also have customer ID.
In the payments table, customer ID links the payment to the customer.

This connection helps build the full business picture.

Customer tells who the person is.
Payment tells what transaction happened.
The foreign key connects them.

Interview answer:
A foreign key is a field that links one table to the primary key of another table.

Analyst answer:
Foreign keys help connect business entities across the system, such as customers, accounts, payments, and transactions.

Rewind:
Foreign keys connect tables.




SECTION: RELATIONAL DATABASE.
A relational database stores data in related tables.

The data is split into separate tables to avoid duplication and improve structure.

Instead of storing all customer, account, payment, and audit information in one huge table, the system separates them.

Customers in one table.
Accounts in another.
Payments in another.
Transactions in another.
Audit logs in another.

Then SQL joins can connect the data when needed.

Interview answer:
A relational database organises data into related tables. Relationships between tables are usually managed using primary keys and foreign keys.

Business answer:
This structure helps keep data cleaner and makes it easier to analyse workflows across the system.

Rewind:
Relational databases split data into connected tables.




SECTION: SELECT.
SELECT is used to retrieve data from a table.

At concept level, SELECT means:
Tell me what information I want to see.

For example:
Show customer ID and customer name.
Show payment ID and payment status.
Show transaction ID, amount, and created date.

In interviews, avoid positioning SELECT as only syntax.
Explain why you use it.

Interview answer:
SELECT is used to retrieve specific columns from a table.

Analyst answer:
I use SELECT to inspect records, validate field values, and understand the current state of the system.

Rewind:
SELECT retrieves data.




SECTION: WHERE.
WHERE filters records based on a condition.

At concept level, WHERE means:
Do not show everything.
Show only records that match this condition.

Examples:
Show only active customers.
Show only failed payments.
Show only transactions above one thousand.
Show records created today.
Show payments for one customer.

This matters because real systems may have millions of records.
An analyst must narrow the result to the affected data.

Interview answer:
WHERE is used to filter rows before results are returned.

Analyst answer:
WHERE helps me focus on affected records during data validation or issue investigation.

Rewind:
WHERE filters data.




SECTION: ORDER BY.
ORDER BY sorts query results.

At concept level, ORDER BY means:
Show the data in a useful order.

You may sort by created date.
You may sort by amount.
You may sort by customer name.
You may sort by status.

In investigation, sorting by latest date is common.

For example:
If a payment failed, sort by created date descending to see the latest payment attempts.

Interview answer:
ORDER BY is used to sort query results in ascending or descending order.

Analyst answer:
ORDER BY helps me review recent transactions, logs, status changes, and events in the correct sequence.

Rewind:
ORDER BY sorts results.




SECTION: COUNT.
COUNT returns the number of records.

At concept level, COUNT answers:
How many?

How many failed payments?
How many active customers?
How many transactions happened yesterday?
How many records have missing email addresses?
How many duplicate customer IDs exist?

COUNT is very useful for business impact.

One failed payment may be minor.
Ten thousand failed payments may be a production incident.

Interview answer:
COUNT is used to count the number of rows matching a condition.

Analyst answer:
COUNT helps me quantify the scale and business impact of an issue.

Rewind:
COUNT measures volume.




SECTION: GROUP BY.
GROUP BY summarises data by category.

At concept level, GROUP BY answers:
How many in each group?

Examples:
How many payments by status?
How many transactions by currency?
How many tickets by priority?
How many customers by account status?

In fintech, GROUP BY is useful for dashboards, operations monitoring, and reporting checks.

Interview answer:
GROUP BY groups rows with the same values and allows aggregate calculations like count or sum.

Analyst answer:
GROUP BY helps me summarise patterns across business categories, such as payment status or transaction type.

Rewind:
GROUP BY creates summaries by category.




SECTION: HAVING.
HAVING filters grouped results.

This is different from WHERE.

WHERE filters individual rows before grouping.
HAVING filters groups after aggregation.

Example:
Find customers with more than three failed payments.

First, filter failed payments.
Then group by customer.
Then use HAVING to show only customers where failed count is greater than three.

Interview answer:
WHERE filters rows before aggregation. HAVING filters grouped results after aggregation.

Analyst answer:
HAVING is useful when I need to find groups that meet a certain threshold, such as customers with multiple failed payments.

Rewind:
HAVING filters after grouping.




SECTION: JOIN.
JOIN connects data from different tables.

This is one of the most important SQL concepts.

Business information is usually split across multiple tables.

Customer details may be in customers.
Account details may be in accounts.
Payment status may be in payments.
Settlement details may be in settlements.
Action history may be in audit logs.

To understand the full workflow, we join tables.

Interview answer:
A join combines related records from two or more tables based on a matching column.

Analyst answer:
I use joins to connect business entities and verify whether the system flow is complete.

Rewind:
JOIN connects related tables.




SECTION: INNER JOIN.
INNER JOIN returns only matching records from both tables.

Example:
Customers with payment records.

If a customer has no payment record, they will not appear in the inner join result.

Business use:
Use INNER JOIN when you only need records where the relationship exists on both sides.

Examples:
Payments with customers.
Accounts with customers.
Transactions with accounts.
Settlements with payments.

Interview answer:
INNER JOIN returns records that have matching values in both tables.

Rewind:
INNER JOIN shows matches only.




SECTION: LEFT JOIN.
LEFT JOIN returns all records from the left table and matching records from the right table.

If there is no match on the right side, the right-side fields return NULL.

This is very useful for finding missing records.

Examples:
Payments without settlements.
Customers without accounts.
Transactions without audit logs.
Applications without status updates.

Interview answer:
LEFT JOIN returns all records from the left table, plus matching records from the right table. If no match exists, the right-side columns show NULL.

Analyst answer:
LEFT JOIN is useful for reconciliation and missing-data investigation.

Rewind:
LEFT JOIN helps find missing relationships.




SECTION: NULL.
NULL means missing, unknown, or not available.

NULL is not zero.
NULL is not an empty string.
NULL means no value is stored.

Examples:
Email is NULL.
Settlement date is NULL.
Approval timestamp is NULL.
Audit user is NULL.

Why it matters:
Missing email may affect customer communication.
Missing settlement date may suggest payment is not settled.
Missing audit user may affect compliance evidence.

Interview answer:
NULL represents a missing or unknown value. It must be checked using IS NULL or IS NOT NULL.

Analyst answer:
NULL checks are important because missing data can affect workflows, reporting, compliance, and customer experience.

Rewind:
NULL means missing or unknown.




SECTION: DISTINCT.
DISTINCT returns unique values.

At concept level, DISTINCT answers:
What unique values exist?

Example:
What payment statuses exist in the system?

Expected statuses may be pending, completed, failed, cancelled.

But DISTINCT may reveal inconsistent data:
FAILED,
Failed,
FAIL,
Failure.

This can expose data quality problems.

Interview answer:
DISTINCT removes duplicate values from the result set.

Analyst answer:
DISTINCT helps me understand unique categories and detect inconsistent values.

Rewind:
DISTINCT shows unique values.




SECTION: CASE.
CASE adds conditional logic in SQL.

At concept level, CASE means:
If this condition is true, return this result.
Otherwise return another result.

Example:
If amount is above ten thousand, mark high value.
If amount is above one thousand, mark medium value.
Otherwise mark low value.

Business use:
CASE can help classify records based on business rules.

Interview answer:
CASE is used to apply conditional logic within a query.

Analyst answer:
CASE is useful for report logic, data categorisation, and business rule validation.

Rewind:
CASE adds if-then logic.




SECTION: SUBQUERY.
A subquery is a query inside another query.

At concept level, it means:
Use the result of one query inside another query.

Example:
Find customers whose customer IDs appear in failed payments.

The inner query finds customer IDs from failed payments.
The outer query retrieves customer details.

Interview answer:
A subquery is a nested query used to filter or calculate results based on another query.

Analyst answer:
Subqueries are useful when the analysis needs multiple logical steps.

Rewind:
A subquery is a query inside a query.




SECTION: DATE FILTERING.
Date filtering is used to retrieve records from a specific time period.

Examples:
Transactions from today.
Payments from last week.
Tickets created this month.
Audit logs after a release.

Date filtering matters because many investigations are time-based.

If a production issue started after a release, you may check records after that release time.

Interview answer:
Date filtering helps retrieve records within a specific date or timestamp range.

Analyst answer:
I use date filtering to investigate issues within a release window, reporting period, or incident timeframe.

Rewind:
Date filtering narrows data by time.




SECTION: DUPLICATES.
Duplicate records occur when the same business record appears more than once when it should not.

Examples:
Duplicate customers.
Duplicate payments.
Duplicate transactions.
Duplicate support tickets.

Duplicates can cause reporting errors, customer confusion, compliance issues, or payment problems.

SQL can detect duplicates using GROUP BY and HAVING.

Interview answer:
Duplicates can be identified by grouping records by key fields and checking where the count is greater than one.

Analyst answer:
Duplicate checks are important for data quality, reporting accuracy, and operational reliability.

Rewind:
Duplicates create data quality risk.




SECTION: RECONCILIATION.
Reconciliation means comparing two sources to confirm they match.

In fintech, reconciliation is critical.

Example:
A payment exists in the payments table.
A matching settlement should exist in the settlements table.

If the payment exists but settlement is missing, there may be a processing gap.

SQL helps compare source and target data.

Business examples:
Payment versus settlement.
Order versus invoice.
Transaction versus ledger.
Report value versus source data.

Interview answer:
Reconciliation is the process of comparing records between systems or tables to confirm completeness and accuracy.

Analyst answer:
SQL helps me identify missing, mismatched, or inconsistent records during reconciliation.

Rewind:
Reconciliation checks whether records match.




SECTION: AUDIT LOGS.
Audit logs record system actions.

They answer:
Who did something?
What did they do?
When did they do it?
Which record was affected?

In regulated financial systems, audit logs are important because they provide evidence.

Examples:
Who changed customer address?
Who approved a payment?
Who cancelled a transaction?
When was the status updated?

Interview answer:
Audit logs provide a history of user or system actions, which supports investigation, traceability, and compliance.

Analyst answer:
I use audit logs to verify whether important actions were captured correctly.

Rewind:
Audit logs provide traceability.




SECTION: UAT.
UAT means User Acceptance Testing.

The purpose of UAT is to confirm that the system works for real business users and meets business requirements.

SQL supports UAT by checking backend data.

The user interface may show a payment as completed.
But SQL can confirm whether the database stored the correct status, amount, timestamp, and audit trail.

Interview answer:
In UAT, SQL helps validate that the backend data matches expected business outcomes.

Analyst answer:
I use SQL during UAT to confirm that system changes are reflected correctly in the database.

Rewind:
SQL supports UAT by validating backend data.




SECTION: DEFECT INVESTIGATION.
Defect investigation means understanding why something went wrong.

SQL helps answer:
What happened?
Which records are affected?
When did it happen?
Is it one user or many users?
Is data missing, duplicated, delayed, or incorrect?
Did the status change correctly?
Is the issue in source data, processing logic, target data, or audit logs?

Interview answer:
SQL helps investigate defects by checking affected records, statuses, timestamps, and relationships between tables.

Analyst answer:
SQL lets me move from guessing to evidence-based investigation.

Rewind:
SQL helps find evidence.




SECTION: PERFORMANCE AWARENESS.
At analyst level, you do not need to sound like a database administrator.

But you should show basic performance awareness.

Good habits:
Do not always use SELECT star.
Retrieve only needed columns.
Use WHERE filters.
Limit large data pulls.
Use IDs, status, and dates carefully.
Ask developers or DBAs for help with complex performance tuning.

Interview answer:
I write targeted queries and avoid unnecessary data extraction. For deeper performance tuning, I would work with database specialists.

Rewind:
Good SQL is focused and controlled.




SCENARIO PRACTICE BLOCK.


SCENARIO DRILL: Failed payment investigation.

Imagine a business user reports that a payment failed.

Your SQL thinking should be:
Find the payment record.
Check payment ID, customer ID, amount, status, created date, and updated date.
Check whether the customer exists.
Check whether an account exists.
Check whether a settlement record exists.
Check audit logs for status changes.
Count how many similar failures happened.

Interview phrasing:
I would first identify the affected payment record, then validate the status, timestamps, customer relationship, settlement record, and audit trail. I would also check whether the issue is isolated or affecting multiple records.

Rewind:
For failed payments, SQL helps identify record, status, relationship, audit trail, and impact.


SCENARIO DRILL: Missing settlement.

Imagine payment status is completed, but settlement is missing.

Your SQL thinking should be:
Payments table is the source.
Settlements table is the target.
Use a LEFT JOIN from payments to settlements.
Find payments where settlement ID is NULL.
Filter by date or payment status.
Measure the count.
Report business impact.

Interview phrasing:
I would compare completed payments against settlement records using a left join to identify payments without matching settlements. Then I would quantify the number of affected records and share examples with the technical team.

Rewind:
Missing settlement is a reconciliation problem.


SCENARIO DRILL: Duplicate customer records.

Imagine a customer appears twice in the system.

Your SQL thinking should be:
Identify possible duplicate fields.
Group by email, phone, or customer reference.
Use count greater than one.
Check created dates.
Check related accounts and payments.
Assess business impact.

Interview phrasing:
I would use grouping and count checks to identify duplicate customer records, then review related account or payment records to understand operational impact.

Rewind:
Duplicate records create data quality and reporting risk.


SCENARIO DRILL: UAT validation.

Imagine a new feature changes payment status from pending to completed.

Your SQL thinking should be:
Before test, capture initial record state.
Perform the action in the application.
After test, query the database.
Check status, updated timestamp, amount, customer ID, and audit log.
Compare expected result against actual result.

Interview phrasing:
During UAT, I would validate not only the screen behaviour but also the backend data to confirm the correct status, timestamp, and audit trail were captured.

Rewind:
UAT should validate both UI and database outcome.


SCENARIO DRILL: Report mismatch.

Imagine a dashboard shows 200 failed payments, but operations expected 180.

Your SQL thinking should be:
Understand report logic.
Check source table.
Filter by same date range.
Filter by same status values.
Check whether duplicates exist.
Check whether cancelled or reversed payments are included.
Check timezone or timestamp rules.
Compare SQL result against report result.

Interview phrasing:
I would first understand the report definition, then recreate the logic using SQL to validate filters, status values, date ranges, duplicates, and inclusion rules.

Rewind:
Report mismatch often comes from filter, date, status, or duplicate logic.



INTERVIEW ANSWER PRACTICE BLOCK.


Interviewer asks: How strong are you in SQL?

Raj answer:
I am comfortable with SQL for practical system analysis work.

I use SQL to retrieve records, filter data, join related tables, validate statuses, check missing values, identify duplicates, support UAT, and investigate defects.

My strength is using SQL to connect technical data with business meaning.

For financial systems, that is valuable because accuracy, reconciliation, reporting, and auditability are critical.


Interviewer asks: What is the difference between WHERE and HAVING?

Raj answer:
WHERE filters individual rows before grouping.

HAVING filters grouped results after aggregation.

For example, if I want failed payment records, I use WHERE payment status equals failed.

If I want customers with more than three failed payments, I group by customer and use HAVING count greater than three.


Interviewer asks: What is the difference between INNER JOIN and LEFT JOIN?

Raj answer:
INNER JOIN returns only matching records from both tables.

LEFT JOIN returns all records from the left table and matching records from the right table. If no match exists, the right-side values are NULL.

For analysis, INNER JOIN is useful when I need confirmed matches. LEFT JOIN is useful for finding missing records, such as payments without settlements.


Interviewer asks: How would you use SQL in UAT?

Raj answer:
I would use SQL to validate whether the system change produced the expected backend data.

For example, if a payment should move from pending to completed, I would check the payment status, updated timestamp, amount, customer ID, and audit log after the test.

This confirms that the business outcome is reflected correctly in the database.


Interviewer asks: How would you investigate a production issue using SQL?

Raj answer:
I would first identify affected records using filters such as ID, status, date, or user.

Then I would check related tables using joins.

I would look at timestamps, statuses, missing values, duplicates, and audit logs.

Finally, I would quantify impact using counts and provide evidence to the technical or business team.


Interviewer asks: Why is SQL important for a Business Systems Analyst?

Raj answer:
SQL is important because it helps the analyst verify what is happening inside the system.

Business users may report that data is missing, wrong, delayed, or duplicated.

SQL helps validate the records, understand the cause, measure the impact, and communicate clearly with developers and stakeholders.

Interviewer asks: What is the difference between WHERE and HAVING?

WHERE filters individual rows before grouping.

HAVING filters grouped results after aggregation.

INNER JOIN returns only matching records from both tables.

LEFT JOIN returns all records from the left table and matching records from the right table. If no match exists, the right-side values are NULL.

For analysis, INNER JOIN is useful when I need confirmed matches. LEFT JOIN is useful for finding missing records, such as payments without settlements.

Interviewer asks: What is the difference between WHERE and HAVING?

Raj answer:
WHERE filters individual rows before grouping.

HAVING filters grouped results after aggregation.

For example, if I want failed payment records, I use WHERE payment status equals failed.

If I want customers with more than three failed payments, I group by customer and use HAVING count greater than three.


Interviewer asks: What is the difference between INNER JOIN and LEFT JOIN?

Raj answer:
INNER JOIN returns only matching records from both tables.

LEFT JOIN returns all records from the left table and matching records from the right table. If no match exists, the right-side values are NULL.

For analysis, INNER JOIN is useful when I need confirmed matches. LEFT JOIN is useful for finding missing records, such as payments without settlements.
